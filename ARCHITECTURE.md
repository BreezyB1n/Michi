# Michi Architecture

## Goal

Michi is a Guide Agent product surface. It turns a user's natural-language intent into a guided, explainable, and verifiable path through a complex web workspace.

The current architecture target is a local React workbench plus an injected browser shell proof:

1. Accept a user intent.
2. Clarify the path only when needed.
3. Map the intent to a product capability category.
4. Generate guide steps with purpose and completion checks.
5. Advance through page-check-aware steps.
6. Show recovery steps when the page state blocks progress.

## First Milestone Boundary

The first milestone does not automate a live account or mutate the host workspace. It builds the smallest browser UI that proves the Guide Loop:

- User Intent parsing
- Capability explanation
- Guide Path generation
- Guide Step display
- Runtime Page Understanding through local fixtures
- Completion Check
- Recovery Step

## System Units

### Product Shell UI

The Product Shell UI is the user-facing Michi surface. It represents the future in-page guide overlay while running as a local workbench and injected shell during early milestones.

Responsibilities:

- capture User Intent
- show clarifying questions
- show the current Guide Step
- show Step Purpose and Completion Check
- show page-check signals
- show Recovery Steps when blocked
- keep visible copy in Michi product language

### Guide Agent Core

The Guide Agent Core owns the Guide Loop.

Responsibilities:

- interpret User Intent
- retrieve Product Skill Pack entries
- map intent to Capability
- produce the Guide Path
- choose the next Guide Step based on page state
- request confirmation for Critical Write Actions
- produce Recovery Steps for Blocking States

### Product Skill Pack

The Product Skill Pack is offline knowledge for a supported workspace path.

Responsibilities:

- list relevant Capabilities
- explain product concepts
- define guide path candidates
- mark Critical Write Actions
- define Completion Checks
- describe common Blocking States

### Runtime Page Understanding

The current runtime uses checked page state from local fixtures and injected-shell reads. A later milestone can widen this with DOM, accessibility tree, page text, or visual recognition data.

Responsibilities:

- report current location
- report available actions
- report account or permission signals
- report whether a Completion Check is satisfied
- report Blocking States

## Runtime Flow

1. The user enters a User Intent in the Product Shell UI.
2. The Guide Agent Core checks whether a clarifying question is needed.
3. The Guide Agent Core retrieves Product Skill Pack knowledge.
4. The Guide Agent Core explains the selected Capability.
5. The Guide Agent Core creates a Guide Path.
6. The Product Shell UI displays the current Guide Step.
7. Runtime Page Understanding returns checked state.
8. The Guide Agent Core runs a Completion Check.
9. The Guide Agent Core advances, asks for confirmation, or emits a Recovery Step.

## Non-Goals

- no live write automation in the first milestone
- no arbitrary script execution generated by the agent
- no broad host-workspace coverage
- no custom-domain live-path requirement for the primary guide
